# 步骤 08：格式不匹配的合并保护验证

## 目标
验证 `assert_mergeable` 校验逻辑真正生效——用不同质量档位（不同采样率/声道数）录制的
两段音频去合并时，系统应明确拒绝并返回可读的错误信息，而不是静默生成一个已损坏的
音频文件。

## 涉及文件
无新增文件。复用步骤 04 的 `AudioRecorder`（切换不同 quality 预设）和步骤 07 的合并
调用入口，仅验证异常路径。

## 实现要点
`format.rs` 中的 `assert_mergeable` 已在步骤 07 中被 `merge_wav_files` 调用，本步骤
无需改代码，重点是补齐前端对 `AppError::AudioFormatMismatch` 的错误展示：
```ts
try {
  await mergeTracks([pathLow, pathHigh], outputPath);
} catch (e) {
  // 应能捕获到形如 "音频格式不匹配，无法无损合并: ..." 的字符串
  console.error(e);
  alert(String(e)); // 或用更友好的 UI 提示组件
}
```

## 验证方法
1. 用 Low 预设（16kHz单声道）录一段
2. 用 High 预设（48kHz双声道）录另一段
3. 尝试合并这两段
4. 前端应收到并展示明确的错误提示（包含"格式不匹配"或类似字样），而不是白屏/无响应
5. 检查输出目录，确认没有生成 `output_path` 对应的文件（或即使生成了空/半截文件，也
   应在错误分支里被清理，不遗留半成品）

## 完成标准 (DoD)
- [ ] 合并不一致格式的音频时，前端明确展示错误提示
- [ ] 错误信息可读（能看出是"格式不匹配"而非笼统的 unknown error）
- [ ] 未产生半损坏的输出文件（或产生后被正确清理）
- [ ] 该操作不会导致应用崩溃或需要重启
